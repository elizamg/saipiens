DynamoDB Table Schemas (Backend)

Region: us-west-1
Design approach: Multi-table, contract-aligned, one entity per table.
All primary identifiers use the field name `id` (String) unless otherwise specified.
Non-key attributes are schema-less and may be added at write time.

---

1. Students

---

Purpose:

* Represents an individual student.
* Used for current user resolution and student-scoped data.

Table name:

* Students

Primary key:

* Partition key: id (String)

Attributes (typical):

* id: string (Cognito `sub`)
* name: string
* yearLabel: string
* avatarUrl: string (optional)

Access patterns:

* Get student by id

---

2. Instructors

---

Purpose:

* Represents instructors associated with courses.

Table name:

* Instructors

Primary key:

* Partition key: id (String)

Attributes (typical):

* id: string
* name: string
* avatarUrl: string (optional)

Access patterns:

* Get instructor by id
* Batch get instructors by id list

---

3. Courses

---

Purpose:

* Represents a course containing units.

Table name:

* Courses

Primary key:

* Partition key: id (String)

Attributes (typical):

* id: string
* title: string
* icon: string (optional)
* instructorIds: string[]
* enrolledStudentIds: string[] (optional; enrollment queries handled via Enrollments table)

Access patterns:

* Get course by id

---

4. Enrollments

---

Purpose:

* Maps students to the courses they are enrolled in.
* Enables efficient listing of courses per student.

Table name:

* Enrollments

Primary key:

* Partition key: studentId (String)
* Sort key: courseId (String)

Attributes (typical):

* studentId: string
* courseId: string

Access patterns:

* List courses for a given studentId

---

5. Units

---

Purpose:

* Represents units within a course.

Table name:

* Units

Primary key:

* Partition key: id (String)

Global Secondary Index:

* CourseUnitsIndex

  * Partition key: courseId (String)
  * Sort key: id (String)

Attributes (typical):

* id: string
* courseId: string
* title: string
* status: string (e.g., active | completed | locked)

Access patterns:

* Get unit by id
* List units for a courseId (via CourseUnitsIndex)

---

6. Objectives

---

Purpose:

* Represents learning objectives within a unit.

Table name:

* Objectives

Primary key:

* Partition key: id (String)

Global Secondary Index:

* UnitObjectivesIndex

  * Partition key: unitId (String)
  * Sort key: id (String)

Attributes (typical):

* id: string
* unitId: string
* kind: string (knowledge | skill)
* title: string

Access patterns:

* Get objective by id
* List objectives for a unitId (via UnitObjectivesIndex)

---

7. Questions

---

Purpose:

* Represents individual questions tied to objectives.

Table name:

* Questions

Primary key:

* Partition key: id (String)

Global Secondary Index:

* ObjectiveQuestionsIndex

  * Partition key: objectiveId (String)
  * Sort key: difficultyStars (Number)

Attributes (typical):

* id: string
* objectiveId: string
* difficultyStars: number (1–3)
* prompt: string

Access patterns:

* Get question by id
* List questions for an objectiveId sorted by difficultyStars ascending (via ObjectiveQuestionsIndex)

---

8. StudentObjectiveProgress

---

Purpose:

* Tracks a student’s progress on an objective.

Table name:

* StudentObjectiveProgress

Primary key:

* Partition key: studentId (String)
* Sort key: objectiveId (String)

Attributes (typical):

* studentId: string
* objectiveId: string
* earnedStars: number
* currentQuestionId: string (optional)
* updatedAt: string (ISO timestamp)

Access patterns:

* Get progress for a studentId + objectiveId
* List progress for a student across objectives (query by studentId)

---

9. ChatThreads

---

Purpose:

* Represents a chat thread for a unit/objective.

Table name:

* ChatThreads

Primary key:

* Partition key: id (String)

Global Secondary Index:

* UnitThreadsIndex

  * Partition key: unitId (String)
  * Sort key: objectiveId (String)

Attributes (typical):

* id: string
* courseId: string
* unitId: string
* objectiveId: string
* title: string
* kind: string
* lastMessageAt: string (ISO timestamp)

Access patterns:

* Get thread by id
* List threads for a unitId (via UnitThreadsIndex)

---

10. ChatMessages

---

Purpose:

* Stores individual messages within a chat thread.

Table name:

* ChatMessages

Primary key:

* Partition key: threadId (String)
* Sort key: createdAt (String, ISO timestamp)

Attributes (typical):

* id: string
* threadId: string
* questionId: string (optional)
* role: string (student | assistant | system)
* content: string
* createdAt: string (ISO timestamp)
* attachments: object (optional)
* metadata: object (optional)

Access patterns:

* List messages for a thread sorted by createdAt ascending

---

11. Awards

---

Purpose:

* Represents awards earned by a student.

Table name:

* Awards

Primary key:

* Partition key: studentId (String)
* Sort key: id (String)

Global Secondary Index:

* StudentCourseAwardsIndex

  * Partition key: studentId (String)
  * Sort key: courseId (String)

Attributes (typical):

* id: string
* studentId: string
* courseId: string
* title: string
* subtitle: string
* iconKey: string

Access patterns:

* List awards for a student
* List awards for a student filtered by courseId

---

12. FeedbackItems

---

Purpose:

* Represents feedback items shown to a student.

Table name:

* FeedbackItems

Primary key:

* Partition key: studentId (String)
* Sort key: id (String)

Global Secondary Index:

* CourseFeedbackIndex

  * Partition key: courseId (String)
  * Sort key: unitId (String)

Attributes (typical):

* id: string
* studentId: string
* courseId: string
* unitId: string
* title: string
* body: string
* ctaLabel: string (optional)
* sourceType: string
* instructorId: string (optional)

Access patterns:

* List feedback for a student
* List feedback for a course

---

Notes:

* All non-key attributes are optional at the database level.
* GSIs exist only to support required access patterns; do not index display-only fields.
* Sorting behavior is guaranteed only when using sort keys (table or GSI).
* ISO 8601 timestamps are used consistently for ordering and debugging.
